### ホワイトボックステスト

* ホワイトボックステストは、コードの中身を見てテストするため実装ミスを見つけやすく、カバレッジも測定できるが、本来必要な機能が実装されていない場合は見逃すことがある

#### ステートメントテスト

* コードの各ステートメント（実行可能な行）を実行することを確認するテスト技法

例）

```java
if(age > 18){
    discount = true;
}
```

テストデータ

```text
age = 20
```

↓

```java
if(age > 18){
    discount = true;
}
```

を実行できるため、ステートメントカバレッジは100%になる

ただし、

```java
if(age >= 18)
```

が正しい仕様なのに

```java
if(age > 18)
```

と誤実装されていた場合、

```text
age = 20
```

では欠陥を発見できない

→ ステートメントを実行したことは確認できるが、正しく実装されていることまでは保証できない

#### ブランチテスト

* コード内のすべてのブランチ（真・偽の判定結果）を実行することを確認するテスト技法

例）

```java
if(age >= 18){
    discount = true;
}
```

ブランチ

```text
真：age >= 18
偽：age < 18
```

テストデータ

```text
age = 20
age = 10
```

↓

真と偽の両方を実行できるため、ブランチカバレッジは100%になる

ただし、

```java
if(A){
    処理1;
}

if(B){
    処理2;
}
```

の場合、

```text
TC1：A=true、B=false
TC2：A=false、B=true
```

でブランチカバレッジ100%になるが、

```text
A=true、B=true
```

の時だけ発生する欠陥は検出できない可能性がある

→ ブランチを実行したことは確認できるが、すべてのパス（組み合わせ）を確認したことにはならない

#### カバレッジの関係

* ブランチカバレッジ100%

  * ステートメントカバレッジ100%

* ステートメントカバレッジ100%

  * ブランチカバレッジ100%ではない

### 経験ベースのテスト技法

経験ベースのテスト技法は、テスト担当者の経験や知識を利用してテストケースを作成する技法である

##### エラー推測法

* アプリケーションの過去の動作状況、開発担当者が起こしやすいエラー、類似アプリケーションで発生した故障などの経験や知識を基に、欠陥がありそうな箇所を予測してテストする

###### メリット

* 経験を活用して効率よく欠陥を発見できる

###### デメリット

* テスト担当者の経験や知識に大きく依存する

###### フォールト攻撃

* エラー推測を体系的に行う方法であり、想定されるエラー、欠陥、故障のリストを基にテストケースを設計する

##### 探索的テスト

* テスト対象について学習しながら、テストケースを同時に設計、実行、評価し、まだテストしていない領域を探索する

* 探索的テストは、仕様がほとんどない場合や不十分な場合、テストのスケジュールに余裕がない場合に有効であり、他のテスト技法（同値分割法など）と併用することもできる。また、経験、ドメイン知識、分析力、好奇心、創造性を持つテスト担当者ほど効果が高い

###### セッションベースドテスト

* 探索的テストを体系的に行う方法であり、あらかじめ決めた時間枠（テストセッション）の中で、テストチャーター（テスト目的や重点的に確認する内容を記載したもの）に従って実施する。テスト担当者は実施内容や発見事項を記録し、終了後にデブリーフィング（振り返り）で結果を共有する

##### チェックリストベースドテスト

* 経験や過去の障害を基に作成したチェックリストを利用してテストする技法

* チェックリストは、経験、ユーザーにとって重要なこと、ソフトウェアが不合格になる理由や仕組み（過去の障害や発生しやすい欠陥など）を基に作成する
* チェックリストには、自動的に確認できる項目、開始・終了基準に関する項目、正常に動作することなどの一般的すぎる項目は含めない方がよい
    * 理由 ＝人が確認する価値が低い、または抽象的で具体性がなく直接チェックできないため

###### メリット

* 詳細なテストケースがなくてもテストのガイドラインとなり、抜け漏れ防止や一定の品質確保に役立つ

###### デメリット

* チェックリストが大まかだとテスト担当者によって実施内容にばらつきが生じ、再現性が低くなる場合がある

* チェックリストは欠陥分析を基に定期的に更新する必要がある

###### 更新が必要な理由

* 開発担当者は同じエラーを避けることを学ぶため、既存のチェック項目の効果が薄れることがある。また、新たに発見した重要度の高い欠陥を反映する必要があるため


### コラボレーションベースのテストアプローチ

* コラボレーションベースのテストアプローチは、欠陥を発見するだけでなく、関係者間（顧客、開発担当者、テスト担当者）のコミュニケーションによって欠陥を未然に防ぐことを目的とする

#### ユーザーストーリー

* ユーザーにとって価値のある機能を表す
* ユーザーストーリーの共同執筆では、コラボレーションによって、ビジネス、開発、テストの視点を考慮して共通のビジョンを得ることが重要

* ユーザーストーリーには「3つのC」がある

  * Card（カード）

    * ユーザーストーリーを記述する媒体

  * Conversation（会話）

    * 関係者間で機能の詳細を議論すること

  * Confirmation（確認）

    * ユーザーストーリーの受け入れ基準

* 良いユーザーストーリーは INVEST を満たす

  * Independent（独立している）
  * Negotiable（交渉可能である）
  * Valuable（価値がある）
  * Estimable（見積り可能である）
  * Small（小さい）
  * Testable（テスト可能である）


#### 受け入れ基準

* ユーザーストーリーの実装をステークホルダーが受け入れるために満たさなければならない条件

* 受け入れ基準の用途

  * ユーザーストーリーのスコープを定義する
  * ステークホルダー間の合意形成を行う
  * ポジティブシナリオとネガティブシナリオを明確にする
  * 受け入れテストのベースとなる
  * 正確な計画や見積りに貢献する

* 受け入れ基準の記述方法

  * シナリオ指向（Given / When / Then）
  * ルール指向（箇条書きや表形式）

#### ATDD（受け入れテスト駆動開発）

* テストファーストのアプローチであり、ユーザーストーリーを実装する前にテストケースを作成する

* 顧客、開発担当者、テスト担当者が協力してユーザーストーリーや受け入れ基準を分析し、テストケースを作成する

* 最初に仕様化ワークショップを行い、ユーザーストーリーや受け入れ基準の不完全さ、曖昧さ、欠陥を解消する

* テストケースは受け入れ基準に基づいて作成し、ユーザーストーリーの実例として利用する

  * テストケースを先に作成することで、チームがユーザーストーリーを正しく理解し、正しく実装するのに役立つ

* 一般的に、ポジティブテスト → ネガティブテスト → 非機能テストの順で検討する
